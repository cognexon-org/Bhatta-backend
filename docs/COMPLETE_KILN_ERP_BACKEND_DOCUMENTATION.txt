1. PROJECT OBJECTIVE
====================

MunimSarthi is a complete Brick Kiln / Bhatta ERP backend system designed to manage the full operational and financial lifecycle of a brick manufacturing business.

The ERP should support:

- Admin Web Panel
- Manager / Munshi Mobile App
- Multi-kiln management
- Multi-season accounting
- Worker and attendance management
- Worker peshgi / advance tracking
- Worker ledger and salary calculation
- Brick production process tracking
- Brick stage-wise and quality-wise stock
- Chamber-wise kiln operations
- Coal and fuel purchase/consumption
- Customer, village, beat, and ledger management
- Orders, dispatches, challans, and invoices
- Udhari / credit management
- Payment collection and due reminders
- Expense, purchase, supplier, and cashbook management
- Owner dashboard and reports
- Voice remarks by managers
- Hindi and English support
- Complete activity logs and approval workflows


2. BACKEND GOALS
================

The backend must solve real daily problems of brick kiln owners and managers:

1. How many raw bricks were made today?
2. How many bricks are drying?
3. How many bricks were loaded into the kiln?
4. Which chamber is loaded, firing, cooling, or ready for nikasi?
5. How much coal was consumed?
6. How many finished bricks are available by quality?
7. How much work did each worker do?
8. How much peshgi / advance has each worker taken?
9. How much salary/payment is due to each worker?
10. Which customers owe money?
11. Which payments are due today or overdue?
12. How many bricks were dispatched?
13. What are today's expenses?
14. What is the current stock?
15. What is the estimated profit/loss for the season?

The backend must be ledger-driven, audit-friendly, and safe for financial and stock operations.

4. USER ROLES AND ACCESS CONTROL
================================

4.1 Roles

ADMIN
- Full control over system.
- Can create kilns, seasons, managers, workers, categories, chambers, approvals, reports, and all financial entries.
- Can approve/reject stock corrections, expenses, process corrections, and sensitive changes.

MANAGER / MUNSHI
- Field user.
- Can enter attendance, production process entries, worker advances, customer orders, dispatches, payments, expenses, leads, and voice remarks.
- Can view data only for assigned kiln, assigned villages, assigned workers, and assigned customers.
- Some actions may require admin approval.

WORKER
- Optional future login role.
- Can view own work, attendance, advances, payments, and salary summary.
- In the MVP, worker may remain only a labor record and not a login user.

4.2 Access Control Rules

- Every protected API must require JWT.
- Every user must have an active account.
- Manager can only access assigned kiln data.
- Manager can only access assigned village/customer/worker data.
- Admin can access all assigned organization data.
- Sensitive updates should generate activity logs.
- Approval APIs should be admin-only.
- Deleted records should preferably be soft deleted using isActive=false or deletedAt.

6. BRICK KILN BUSINESS FLOW
===========================

The backend should follow the real bhatta production lifecycle:

Mitti Khudai
  ↓
Raabis / Clay Preparation
  ↓
Pathai / Brick Molding
  ↓
Kaccha Brick Stock
  ↓
Sukhai / Drying
  ↓
Sukhi Brick Stock
  ↓
Bharai / Kiln Loading
  ↓
Chamber Loaded
  ↓
Phukai / Firing
  ↓
Pakai / Baking
  ↓
Cooling
  ↓
Nikasi / Unloading
  ↓
Chhantai / Quality Sorting
  ↓
Finished Stock
  ↓
Order
  ↓
Dispatch / Challan
  ↓
Payment / Udhari / Customer Ledger

Backend principle:

Each major process must create proper records and ledger effects.

Examples:

- Pathai adds Kaccha Eent stock.
- Sukhai transfers Kaccha Eent to Sukhi Eent.
- Bharai moves Sukhi Eent into chamber.
- Phukai records coal/fuel consumption.
- Nikasi removes fired bricks from chamber.
- Chhantai splits bricks into No.1, No.2, No.3, Jhama, Broken, etc.
- Dispatch reduces finished stock.
- Payment updates customer ledger and cashbook.
- Worker work creates worker ledger earning.
- Worker advance/payment updates worker ledger and cashbook.

10. WORKER LEDGER, PESHGI, AND PAYROLL

Purpose:
Track worker earnings, advances, payments, deductions, and final payable balance.

This is one of the most important modules for a bhatta ERP.

10.1 Worker Ledger

Transaction types:

- EARNING
- ADVANCE
- PAYMENT
- DEDUCTION
- ADJUSTMENT
- OPENING_BALANCE

Important fields:

- ledgerId
- workerId
- kilnId
- seasonId
- managerId
- date
- transactionType
- sourceModule
- sourceId
- processCode
- quantity
- unit
- rate
- amount
- debit
- credit
- balanceAfter
- paymentMode
- referenceNo
- remark
- voiceRemarkId
- createdBy
- createdAt

Recommended accounting logic:

- EARNING increases payable amount to worker.
- ADVANCE reduces payable amount or increases recoverable balance.
- PAYMENT reduces payable amount.
- DEDUCTION reduces payable amount.
- ADJUSTMENT can increase or decrease balance depending on correction.

Example ledger:

Opening balance:       0
Pathai earning:        + 2,000
Peshgi advance:        - 1,000
Cash payment:          -   500
Remaining payable:       500

Business rules:

10.3 Worker Payment

Features:

- Payment history

Business rules:

10.4 Payroll

Purpose:
Track clay preparation.

- Water usage
- Labor used
- Area/location
- Remark

11.4 Sukhai Entry

Purpose:
Move raw bricks into dried brick stage.

Input:
- Quantity taken from Kaccha Eent
- Quantity dried
- Damaged/wastage quantity
- Date

Stock effect:
- Reduce KACCHA_EENT
- Add SUKHI_EENT
- Add wastage if tracked

11.5 Bharai Entry

Purpose:
Load dried bricks into kiln chamber.

Input:
- Chamber
- Quantity
- Workers
- Date

Stock effect:
- Reduce SUKHI_EENT
- Increase chamber current quantity

Chamber effect:
- EMPTY -> LOADING
- LOADING -> LOADED when capacity reached or marked completed

Worker effect:
- Create WorkerLedger EARNING

11.6 Phukai Entry

Purpose:
Record firing process and coal/fuel consumption.

Input:
- Chamber
- Start date/time
- End date/time
- Coal/fuel used
- Workers
- Remark

Fuel effect:
- Reduce FuelStock
- Create FuelConsumption record

Chamber effect:
- LOADED -> FIRING
- FIRING -> COOLING after firing completed

Worker effect:
- Create WorkerLedger EARNING if applicable

11.7 Pakai Entry

Purpose:
Track baking progress after firing.

Input:
- Chamber
- Date
- Status
- Remark

Chamber effect:
- FIRING or COOLING status update

11.8 Nikasi Entry

Purpose:
Unload fired bricks from kiln chamber.

Input:
- Chamber
- Quantity unloaded
- Workers
- Date
- Remark

Chamber effect:
- READY_FOR_NIKASI -> COMPLETED or EMPTY depending on workflow

Worker effect:
- Create WorkerLedger EARNING

11.9 Chhantai Entry

Purpose:
Quality sorting after nikasi.

Input:
- Chamber
- Total quantity
- Quality-wise split:
  - NO_1
  - NO_2
  - NO_3
  - JHAMA
  - BROKEN
  - CHHATTA
  - OTHER

Stock effect:
- Add finished stock by category.

Business rule:
- Sum of quality output should not exceed input quantity unless admin override is allowed.
- Difference should be treated as wastage.

36. BUSINESS VALIDATIONS
========================

36.1 Common Validations

- Validate all ObjectIds.
- Validate required fields.
- Validate enum values.
- Validate positive quantity.
- Validate positive amount.
- Validate date formats.
- Validate active kiln and active season.
- Validate manager scope.
- Validate duplicate records.
- Backend must recalculate totals instead of trusting frontend totals.

36.2 Stock Validations

- Quantity must be greater than zero.
- Stock reduction cannot exceed available stock.
- Transfer requires source and destination categories.
- Dispatch can use only saleable stock.
- Stock correction requires reason.
- Manager stock correction should require admin approval.

36.3 Process Validations

- processCode must exist and be active.
- Chamber is required when process requires chamber.
- Workers are required when process requires workers.
- Fuel is required when process requires fuel.
- quantityOut cannot be negative.
- Wastage cannot be negative.
- Chhantai output sum should not exceed input quantity unless override.
- Manager can create process entry only for assigned kiln.

36.4 Worker Ledger Validations

- Worker must be active.
- Amount must be positive.
- Worker advance/payment must create cashbook transaction.
- Worker earning should be linked to valid process entry or attendance.
- Manual adjustment should require reason.

36.5 Order and Dispatch Validations

- Customer must exist and be active.
- Order items cannot be empty.
- Quantity and rate must be positive.
- Dispatch quantity cannot exceed order pending quantity.
- Dispatch quantity cannot exceed available stock.
- Cancelled orders cannot be dispatched.
- Delivered dispatch should not be edited except by admin correction.

36.6 Payment Validations

- Payment amount must be positive.
- Payment cannot exceed pending amount unless overpayment allowed.
- Payment should update udhari if linked.
- Payment should update customer ledger.
- Payment should create cashbook entry.

36.7 Expense Validations

- Expense amount must be positive.
- Approval required based on category/amount rules.
- Rejected expenses should not create cashbook transaction.
- Approved expense should not be edited without correction entry.


37. TRANSACTION SAFETY REQUIREMENTS
===================================

MongoDB transactions must be used for multi-step operations.

Must use transaction for:

1. Pathai entry:
   - Create ProcessEntry
   - Add stock
   - Create StockLedger
   - Create WorkerLedger
   - Create ActivityLog

2. Sukhai entry:
   - Reduce Kaccha stock
   - Add Sukhi stock
   - Create StockLedger transfer
   - Create ProcessEntry

3. Bharai entry:
   - Reduce Sukhi stock
   - Update chamber quantity/status
   - Create ProcessEntry
   - Create WorkerLedger

4. Phukai entry:
   - Create ProcessEntry
   - Reduce FuelStock
   - Create FuelConsumption
   - Create WorkerLedger
   - Update chamber status

5. Chhantai entry:
   - Create ProcessEntry
   - Add multiple finished stock categories
   - Create multiple StockLedger records
   - Update chamber status

6. Dispatch:
   - Validate order
   - Validate stock
   - Create Dispatch
   - Reduce stock
   - Create StockLedger
   - Update order status
   - Create CustomerLedger
   - Create Udhari if unpaid
   - Create ActivityLog

7. Customer payment:
   - Create Payment
   - Update Udhari
   - Create CustomerLedger
   - Update customer outstanding
   - Create CashTransaction

8. Worker advance/payment:
   - Create WorkerLedger
   - Update worker balance
   - Create CashTransaction

9. Fuel purchase:
   - Create FuelPurchase
   - Update FuelStock
   - Create SupplierLedger
   - Create CashTransaction if paid

10. Expense approval:
   - Update expense status
   - Create CashTransaction
   - Create ActivityLog

FINAL SUMMARY
=============

The complete MunimSarthi backend should be designed around these core concepts:

1. Process-driven production
2. Brick stage and quality-wise stock
3. Stock ledger for every stock movement
4. Worker ledger for earnings, peshgi, and payments
5. Chamber-wise kiln operation
6. Fuel/coal tracking and consumption
7. Customer ledger and udhari tracking
8. Dispatch-based stock reduction
9. Cashbook-based financial tracking
10. Reports and dashboards for owner visibility

The most important backend shift is:

FROM:
Simple KACCHI_ETH / PAKKI_ETH production tracking

TO:
Pathai -> Kaccha Stock -> Sukhai -> Bharai -> Phukai -> Pakai -> Nikasi -> Chhantai -> Finished Stock -> Dispatch

This backend structure can support a complete Brick Kiln ERP for real bhatta operations in UP/Bihar and similar regions.
